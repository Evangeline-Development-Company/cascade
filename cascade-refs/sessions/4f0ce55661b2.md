## Overall Impression

This is one of the most thoughtfully designed personal tooling projects I've seen. The design docs read like someone who actually understands their own problem space deeply — not just "I need to apply for scholarships" but "here's exactly how my brain works, here's my writing voice taxonomy, here's why RAG is overkill for 20k tokens of profile data." That level of self-awareness is rare.

## What's Really Strong

**The Story Bank architecture is the star.** The insight that scholarship prompts collapse into ~12 archetypes is correct and underappreciated. Mapping stories to archetypes ahead of time, with strength scores and usage tracking, turns essay writing from "stare at blank page" into "pick from ranked options." The tiered structure (Lead / Strong Supporting / Color) is exactly right — it prevents the trap of reaching for a Tier 3 anecdote when a Tier 1 story fits.

**The voice analysis is genuinely excellent.** The do/don't list, the register range (personal-vulnerable through formal-compressed), the word-count calibration, the specific anti-patterns ("never use the word passion," "don't start with 'Growing up in Phoenix'") — this is a real style guide, not a vague "write like Eve." The note about not mirroring self-deprecation is smart editorial judgment.

**The profile doc is absurdly thorough.** 4 batches of deep-dive data, corrections from Eve herself, name handling rules for different platforms, the SAFE vs UNSAFE mailing address distinction (now updated since coming out). The cross-project pattern analysis at the end ties five seemingly separate technical projects into one coherent narrative. That's the kind of framing that wins engineering scholarships.

**The decision NOT to use RAG is correct.** 20k tokens of structured profile data fits in context. Adding vector embeddings and retrieval infrastructure would be pure complexity theater. The architecture of "Claude Project with structured knowledge + MCP tools for discovery" is the right call.

## Things I'd Push Back On or Flag

**1. Scope creep risk is real.** The MCP server spec describes 6 sprints covering: Bold.org API, ASU cookie auth, external web scraping (Playwright), PostgreSQL tracker, FastAPI dashboard with 5 pages and Swiss Typographic styling, story matching with internal Claude API calls, draft versioning, recycling detection, calendar view, and a stretch goal of a browser bookmarklet. That's a substantial product for someone whose primary goal is *writing scholarship essays before deadlines*. The danger is spending more time building the tool than using it.

My suggestion: ruthlessly prioritize. The story bank YAML, the system prompt, and the Claude Project setup (Phases 1-2 of the copilot spec) deliver 80% of the value in a day. The tracker could be a simple JSON file or even a Notion board. The Bold.org API integration is nice but Eve can also just... browse Bold.org. The dashboard with PCB-trace-copper timeline animations is beautiful design but it's polish on a system that doesn't exist yet.

**2. The ASU cookie auth is fragile by design.** 20-minute session cookies that need manual refresh are a maintenance headache. Unless there's a headless browser flow to re-authenticate, this will break constantly and Eve will stop using it. I'd deprioritize this behind everything else and treat it as optional.

**3. The `match_stories_to_prompt` tool with internal Anthropic API calls is an interesting choice.** Using Claude to rank stories inside an MCP tool that's being called by Claude creates a nested LLM pattern. It works, but it adds latency, cost, and a failure mode. The archetype-based matching (first pass, no API call) is probably sufficient for 90% of prompts — a simple keyword/tag match against 15-20 stories doesn't need semantic understanding.

**4. The style guide is gorgeous but premature.** "The Semiconductor Atelier" with Playfair Display headers, PCB trace copper accents, urgency blur effects, and honeycomb story heatmaps — this reads like a design portfolio piece. It's genuinely cool, but it's a dashboard for a system that needs to be built first. I'd ship a plain HTML table with deadline sorting before thinking about HTMX fade transitions.

**5. Missing: a prioritized scholarship list.** The spec calls this out in "Open Questions for Eve" but it's the most important thing. All the infrastructure in the world doesn't matter without a concrete list of "these 10 scholarships, these deadlines, these prompts." That should be step zero.

## Architecture Thoughts

The overall architecture (FastMCP server + PostgreSQL + FastAPI dashboard + Claude Project for drafting) is sound. The project structure is clean and well-organized. A few specific notes:

- **PostgreSQL over SQLite is the right call** if the dashboard and MCP server access the DB concurrently
- **The schema design is solid** — the `scholarships -> requirements -> drafts` hierarchy with a separate `story_usage` junction table is normalized correctly
- **The sprint ordering makes sense** — foundation first, then discovery, then dashboard, then integration
- **Environment variable management is properly scoped** — no hardcoded secrets in the spec

## What I'd Build First

If I were starting this today, in order:

1. **story-bank.yaml** — Generate the structured story bank from the profile doc. This is immediately useful even without any server.
2. **Claude Project setup** — System prompt with voice guide + story bank as project knowledge. Eve can start drafting essays *today* with this.
3. **A simple scholarship list** — Even a markdown file with name, deadline, status, prompt. No database needed yet.
4. **The tracker tools** (PostgreSQL + basic CRUD) — Once there are 10+ scholarships to track.
5. **Bold.org integration** — Once manual browsing becomes a bottleneck.
6. **Dashboard** — Once there's enough data to visualize.

This isn't just a dev spec. It's someone building a tool that actually understands its user. That's the difference between a project that gets used and one that sits in a repo.
